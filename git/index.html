intro to VCS

Version Control Systems (VCS)

  • wise programmers use a VCS in code development
  • tracks and provides control over changes of data
  • more importantly helps collaboration!
  • storing versions of code
  • restoring code to a previous version
  • comparing changes over time
  • understanding what happened
  • see who to blame for a bug
  • supports the consistency & integrity of data
  • provides a backup of your work

history       

Before-VCS epoch

  • Lower paleolithic: Keep your own versions
  • Stone-age: Use diff, emails and tars

Centralized VCS (CVCS) epoch

  • Bronze age: CVS, a wrong intro
  • Iron age: SVN, perforce

Distributed VCS (DVCS) epoch

  • Modern era: bazaar, arch, monotone, darcs, bitkeeper

Linus and the war of (D)VCS

  • Linus was not happy with any of the OS solutions!
  • He stepped out of Linux development for a month to create his second child, git
  • mercurial, another child of the war
  • git wins the war, becomes the standard in OS community, followed by mercurial

DVCS vs CVCS

Centralized

  • Only a single copy of the repository (unsafe)
  • Server communication for each operation (slow, needs net)
  • Server is offline or corrupted; no work done (offline cost)
  • Everything is stored to the server; even failed things (redundacy)
  • Pros: Rather simple interaction (simple but not flexible)

Distributed

  • More powerful and flexible workflows (promote collaboration)
  • Each user has its own repository (safety)
  • Operations are done locally (performance)
  • Branches is the way to go (easy to create, delete and merge)
  • Cons: Initial checkout is more costly (download the whole history)
  • Cons: A little more difficult to grasp UI (interaction overhead)

your first repo

git init

  • You can initialize a repo with git init
  • $ git init tutorial
  • A new directory tutorial is created
  • Inside tutorial, .git directory holds all repository information

git status

  • At any time you can check your status with git status
  • Rather informative command, but not after init!
  • $ git status
    On branch master
    Initial commit
    nothing to commit (create/copy files and use "git add" to track)

3 trees +  

three trees

  • working directory → sandbox, holds the files
  • index → snapshot of changes proposed for next revision
  • HEAD → pointer to last snapshot
  • init also creates master default & active branch
       detached mode → no active branch
  • HEAD (usually) points where the branch points

create our first file

  • $ touch first
  • File is added to working directory

git add [<file>]

  • Add content to the staging index
  • $ git add first
  • File added to index

logcommit(revision)     

git commit [<file>]

  • Stores the contents of the index in a new commit object
  • along with who, when, and why made those changes
  • parent of commit object is the previous commit
  • Creates a SHA-1 20 byte hash using the above
  • HEAD and master point to new commit
  • $ git commit first
  • Add & commit the file at the same time
  • $ git commit -a first

git log

  • $ git log --decorate --graph
    * commit 8484f4fccb80e625b7c05cdc214014a32d176777 (HEAD)
    | Author: Panagiotis Papadakos <papadako@gmail.com>
    | Date: Tue Oct 20 01:15:40 2015 +0300
    |   Add some content

Grow branches        

git branch <name>

  • Branching is a very cheap and widely used operation
  • Develop features/correct bugs/experiment in isolation
  • ... and delete them without affecting the main history
  • Just creates a new pointer where HEAD points
  • $ git branch featureX && git branch bugY
  • Only one activeBranch (the rest notActiveBranches)
  • master is the default branch (commit still updates the master pointer)

git checkout [<name>]

  • Switch to another branch
  • Updates files in the working tree to match the version
        in the index or the specified tree
  • <name> a branch name or commit object (if not given, HEAD)
  • If modifications cannot be applied cleanly
       ...git won't switch to another branch (use git commit/git stash)
  • $ git checkout featureX

merge-α

git merge <commit>

  • Easy branching without easy merging has no value!
  • Join two or more development histories (branches) together
  • Example: Made a number of commits in branch featureX
  • Incorporate them in master by:
         git checkout master
         git merge featureX
  • The simplest of all merges: fast forward !

merge-β

git merge <commit>

  • A more difficult case!
  • Make a number of commits to featureX
  • Then go to master
  • Make a number of commits
  • Merge master with featureX
  • If branches haven't modified the same lines, git has no problem!
  • A commit object can have many parents
  • Even if we can fast forward git merge -no-ff will create a new commit

conflicts-γ

Merging can lead to conflicts

  • git tries hard and is smart enough in avoiding them
  • But when conflicts occur, no need to be scared!
  • Let them fight and choose the winner
        i.e. review the code and resolve them (removing >>>===<<<)
  • git add resolved files and then commit
    <<<<<<< HEAD
    I am the bug!
    =======
    No, I am the bug!
    >>>>>>> master
    Resolve bug

On-line repos!

git remote

  • The whole thing is about collaboration!
  • git does not have a central repository like SVN
  • Repositories have to be uploaded for sharing
  • All repositories are equal
  • Isolated environment - manual synchronization
  • Supported protocols - http, https, git, filesystem
  • Creates pointers like branch, pointing to other repo's commit objects
  • git remote - Create connections & give nick names to online repos
  • $ git remote add forU https://github.com/papadako/git-a-little-tale.git
  • Remote pointers are updated when we have network traffic

     clone repos, !  

git clone [<repoName>]

  • Clones a repo locally, into a newly created directory
  • Creates remote branches for each branch in the cloned repo
  • Default nick name of remote is origin (like master for init)
  • Checkout an initial branch (cloned repo’s currently active branch)
  • $ git clone https://github.com/papadako/git-a-little-tale.git






, fetch it!

git fetch [<remote>]

  • Brings commits from online repo to local
  • Does not merge the changes though
  • Integration of changes: a two step process
  • $git fetch origin (fetches all online branches of remote origin)
  • $git fetch origin bname (fetches only origin/bname)









Now, pull!       

git pull [<remote>]

  • Merging upstream changes into a local repository is a common task
  • pull makes this two step process in a single step
  • $ git pull is the same as:
         $ git fetch <remote>
         $ git merge origin/<current-branch>
  • Can lead to conflicts as any other merge









Push, push!       

git push [<remote>][<branch>]

  • Publish all commited local changes to a central repository
  • Creates/updates a local branch in the destination repository
  • Tags are not automatically sent (use --all or --tags flags)
  • If non-fast-forward merge, push is not permitted (will change commits)
  • Use --force if you really need this CAUTION









dangerrr!!!       

git rebase (harder collaboration)

  • User can change the history
  • Useful for non-pushed commits only, else collaboration is harder

git reset [<name>](lose commits)

  • Move branch, HEAD to specific commit
  • index and working directory do not change

git rm (lose code)

  • Remove file from index or from index and working directory

git reset --hard [<name>]
(lose code + commits)

  • Move branch, HEAD, index and working directory to specific commit

et al.  

git tag

  • Add a label in a commit object
  • git tag -a v4.4.4 -m 'my lucky version'

git stash

  • If active branch is dirty and have to quickly checkout another branch
  • git stash adds the modified files and staged changes in a stack
  • git stash pop re-inserts the files to the current branch

git bisect

  • Use binary search to find which commit introduced a bug

git revert

  • Revert a commit and commit the change

git mv, show, cherry pick, blame, etc.


There is no file lock command, since there is no server!

  • No locking, means better parallelism & collaboration!

tips

for you, not for the waiter!

  • git tracks content not files
  • be careful with git mv (commit both the old and new file)
  • commit often (easier code review, bug tracking and merging)
  • pull master often (reduces conflicts)
  • keep binary files in another repository (overhead per each clone)
  • make sure commits pass tests and compile cleanly (helps collaboration)
  • delete branches that you are sure are unneeded! (branch overhead)
  • check which files you are committing (stop poisoning history)
  • do not commit passwords (they will always be in the history)
  • when changes are pushed, you can't take them back (almost)
  • be careful about code formatting
       → changes are hidden in noise
       → produce conflicts
       → make explicit commits for formatting

workflows

Centralized (SVN like)

  • Someone initializes a central branch (master)
  • Everybody clones the central repository
  • Each one works to the code (edit, stage, commit)
  • Upload work in master git push origin master
  • If trees have diverged, you have first to git pull
